# Media Dock 3.0 release gates

Stable release requires every blocking gate below to have repeatable evidence from the candidate release assets.

| Gate | Required evidence |
| --- | --- |
| Core domain | Automated task-state, Task Plan, scheduler, recovery, SQLite migration, naming, overwrite, and IPC-contract tests |
| Real media | Public network acquisition and merge, local-media processing, and representative audio or subtitle deliverables |
| Windows | CJK and spaced paths, directory selection and reveal, Cookie-package import, managed-runtime update and rollback, complete process-tree cancellation |
| Windows display | 100%, 125%, 150%, and 200% scaling without root horizontal overflow or unreachable primary action |
| macOS | Candidate DMG and ZIP launch, directory permission, real task, deliverable use, and diagnostic export |
| Data isolation | 2.1.2 remains independently usable; copy import leaves legacy data untouched; failed 3.0 migration restores safely |
| Managed runtimes | Verified staging, atomic activation, failed-download isolation, previous-version rollback, and pinned running-task behavior |
| Product UI | Simplified Chinese and English, light and dark themes, keyboard operation, reduced motion, and four-space core journeys |
| Release artifacts | Installed or extracted GitHub candidate assets report the intended version and match published SHA-256 checksums |

Automated builds do not replace target-machine verification. Authenticated source checks use user-controlled local packages and never place secrets in CI. A failed or missing blocking gate keeps the candidate in Pre-release status.
